Xon Developer Docs + Playground

Xon (Xerxis Object Notation) is JSON-compatible configuration with optional runtime expressions. This page is the product documentation and playground for v1.0.0, including production setup steps for npm projects and web frameworks.

npm: @xerxisfy/xon Node 18+ Build to JSON CLI + Native Addon Trusted Config Runtime 
{
    let host = "0.0.0.0",
    let port = 8080,
    service: {
        host: host,
        port: port,
    },
}

1) What Developers Should Expect in v1.0.0

Xon v1.0.0 is production-ready for trusted, version-controlled configuration files. The npm package gives you a native parser/runtime and CLI for parse, validate, format, convert, eval, and build-time JSON output.

Stable Today npm package @xerxisfy/xon, Node 18+, native addon, and CLI workflows.
Best Production Pattern Keep .xon in source control, validate in CI, and build to JSON for frontend apps.
Safety Note Use runtime eval for trusted inputs unless you isolate execution and apply limits.

2) Install + Verify (Node 18)

Use these exact commands in a new or existing project.

export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && . "$NVM_DIR/nvm.sh"
nvm use 18

npm install @xerxisfy/xon
npx xon --help

Minimal parse usage in Node

const xon = require("@xerxisfy/xon");
const cfg = xon.xonifyString('{ app: "api", port: 8080 }');
console.log(cfg.port); // 8080

3) Build-Time JSON Conversion (Recommended)

Browsers and frontend frameworks do not natively parse .xon files like JSON. The reliable production flow is to compile Xon to JSON during build. 

# Default convention:
# config/app.xon -> src/generated/app-config.json
npx xon build

# Single-file explicit conversion
npx xon build config/app.xon src/generated/app-config.json

# Directory conversion (recursive)
npx xon build config src/generated
Guarantee: generated output is standard JSON, so your app/framework reads the JSON file the same way it reads any hand-written JSON file.

4) Zero-Arg Build With xon.config.json

Add a project-level config once, then teams can run xon build without passing paths. 

{
  "build": {
    "input": "config",
    "output": "src/generated"
  }
}

Save it as xon.config.json in your project root, then run:

npx xon build

CLI arguments still override config when needed: npx xon build config/app.xon src/generated/app-config.json

5) Framework Usage (Vite/Next/Webpack and Others)

Add one build script, commit your .xon files, and import generated JSON in app code.

package.json scripts

{
  "scripts": {
    "xon:build": "xon build",
    "build": "npm run xon:build && vite build",
    "dev": "npm run xon:build && vite dev"
  }
}

App import

import appConfig from "./generated/app-config.json";
// use appConfig anywhere in your app
Important: apps do not consume raw .xon by default. They consume the generated JSON. Running xon build in your pipeline makes this seamless for developers.

8) Expressions, Conditionals, and Variables

Expression support includes precedence-aware arithmetic, comparisons, short-circuit boolean logic, null coalescing, ternary, and expression-style if (...) ... else .... 

{
    let base = 10,
    let rate = 1.25,
    let retries = null,

    subtotal: base * rate + 2,
    effective_retries: retries ?? 3,
    mode: if (effective_retries > 1) "retry" else "single",
    health: subtotal > 10 ? "ok" : "warn",
}
Scope rule: if you want to reuse a value later, declare it with let or const. Plain object fields do not become symbols.

9) Functions (Arrow + Keyword Forms)

Xon supports closures and function aliases: function, func, fun, and fx. 

{
    let tax = 0.16,
    function add(a, b) { a + b },
    func mul(a, b) => a * b,
    fun sub(a, b) => a - b,
    fx add_tax(amount, i) { amount + (amount * tax) },
    let square = function(x) { x * x },

    sum: add(4, 5),
    product: mul(4, 5),
    reduced: sub(10, 3),
    taxed: add_tax(100, 0),
    squared: square(9),
}

11) Safety, Diagnostics, and Type Modes

Runtime execution can be constrained using limits and policies. Diagnostics include line-aware error messages and call-stack traces. 

// C API example
XonEvalOptions options;
xon_eval_options_default(&options);
options.max_call_depth = 512;
options.max_call_count = 50000;
options.max_eval_steps = 250000;
options.max_result_nodes = 100000;
options.duplicate_keys = XON_DUPLICATE_KEY_ERROR;
options.type_mode = XON_TYPE_MODE_STRICT;

XonValue* out = xon_eval_ex(ast, &options);
if (!out) {
    fprintf(stderr, "Eval error: %s\n", xon_get_last_error());
    fprintf(stderr, "Eval stack: %s\n", xon_get_last_error_stack());
}

Key helpers

  • typeof(value), schema_of(value), conforms(value, schema), assert_type(value, expected)
  • symbols() and defined(name) for visibility checks
  • Duplicate key policy and type modes via eval options

13) Works Today vs Not Yet

Works Today (v1.0.0)

  • Parser for JSON-compatible structures with comments, trailing commas, and unquoted keys.
  • Runtime expressions, declarations, functions/closures, conditionals, and built-in helpers.
  • CLI commands: parse, validate, format, convert, eval, build.
  • Build-time conversion workflow for frontend apps via xon build and xon.config.json.
  • Node package with native addon and C API for embedding.

Not Yet (Post v1.0.0 Roadmap)

  • First-class import/include module system with full file-resolution semantics.
  • Built-in schema validation engine.
  • Full advanced literal roadmap (for example binary/octal literal forms).

14) Learning Paths By Experience Level

Beginner Start with object/list syntax, comments, and validate/format. Then add basic let + arithmetic.
Intermediate Add functions, closures, map/filter, templates, and build-time JSON conversion.
Advanced Use custom eval options, strict runtime policies, and app-level schema enforcement after parsing.
